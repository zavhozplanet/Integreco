## 1. СУТЬ ПРОЕКТА

**Integreco** — интерактивная майнд-карта для визуального мышления, построения концептов и графов взаимосвязей.

**Ключевые принципы:**
- Минимум текста в интерфейсе — максимум инфографики (значки, иконки, символы) для универсальной доступности
- Работа на десктопах и тачскринах
- Офлайн-first с локальным хранением
- Вдохновление: Mindomo, Obsidian

**Планы масштабирования:**
- PWA для всех платформ
- Плагин для Obsidian
- **Multi-Nodes (Гибридные блоки):** Масштабируемые узлы с поддержкой произвольных точек подключения на границах и адаптивным поведением текста/скролла.
- **Интеграция с ИИ** (в планах)
- API для сторонних сервисов
- Синхронизация устройств через Syncthing (ближний план) и PouchDB/CouchDB (дальний план)

## 2. СТЕК И АРХИТЕКТУРА

### Текущее состояние
- **Модульная архитектура** — логическая часть разбита на `src/*.js`, стили в `styles/main.css`. `index.html` используется как минимальная оболочка для подключения скриптов в глобальный `window`-контекст.
- **Vanilla JS + SVG**, функциональные модули без бандлеров.
- Узлы — `<div class="node">` с абсолютным позиционированием на `#canvas` (8000×8000px)
- Линии/рёбра — SVG на `#svgl` (8000×8000, поверх canvas, `pointer-events:none`)
- Трансформация: `canvas.style.transform = translate(panX,panY) scale(zoom)`, `transform-origin:0 0`
- Шрифт: Inter (Google Fonts)
- Персистентность: `localStorage`
- PWA: Service Worker (`sw.js`) + `manifest.json`

### Файловая структура проекта
```
Integreco/
├── index.html          # Минимальная оболочка (подключает JS-модули и CSS)
├── styles/
│   └── main.css        # Основные стили проекта
├── src/                # Разделенные логические JS-модули (> 20 файлов)
│   ├── init.js         # Инициализация и автосохранение
│   ├── state.js        # Глобальное состояние (nodes, edges, настройки)
│   ├── geometry.js     # Геометрия Безье рычагов, snapping, render()
│   └── utils.js...     # Функции управления, UI, события мыши/тачскрина
├── sw.js               # Service Worker (v2) для офлайн-кэширования модулей
├── manifest.json       # PWA-манифест
├── package.json        # Конфигурация (ВНИМАНИЕ: содержит React/Tailwind — артефакт AI Studio)
├── vite.config.ts      # Конфигурация Vite (артефакт AI Studio)
├── .env.example        # Пример переменных окружения

```

> ⚠️ **ВАЖНО:** `package.json` содержит зависимости React, Tailwind, motion и т.д., оставшиеся от среды Google AI Studio. Проект НЕ использует React — это чистый vanilla JS. При рефакторинге нужно либо очистить package.json, либо перевести проект на модульную архитектуру.

## 3. СТРУКТУРА ДАННЫХ

### Node Object
```js
{
  id: Number,
  type: 'node' | 'root' | 'note' | 'group' | 'multi',
  x: Number, y: Number,           // координаты центра в пространстве холста
  width: Number, height: Number,   // только для групп
  label: String,
  note: String,                    // текст заметки
  color: String | null,            // цвет фона карточки
  locked: Boolean,                 // блокировка перемещения
  nodes: Array<Number>,            // только для групп — ID узлов внутри
  bg: { color, image, pattern }    // только для групп — индивидуальный фон
}
```

### Edge Object
```js
{
  id: Number,
  from: Number, to: Number,       // ID узлов
  shape: 'straight' | 'bezier' | 'elbow',  // 'curved' в v1 → 'bezier' в v2
  dash: 'solid' | 'dashed' | 'dotted' | 'link',
  width: Number,                   // числовое значение (1, 1.5, 3)
  color: String,
  dir: 'forward' | 'backward' | 'both' | 'none',
  cp1x, cp1y, cp2x, cp2y,         // контрольные точки Безье (null = авто)
  bend: Number,                    // 0.1–0.9, точка изгиба для elbow
  collapsed: Boolean,              // ветка свёрнута
  isLink: Boolean,                 // связующая линия (не иерархическая)
  label: String                    // подпись на линии
}
```

### Глобальное состояние
```
nodes[]              — массив объектов узлов
edges[]              — массив объектов рёбер
idC                  — счётчик ID
zoom, panX, panY     — трансформация вьюпорта
selN                 — id выбранного узла (null если нет)
selE                 — id выбранной линии
selEHandles          — показывать ли рычаги безье
```

## 6. КЛЮЧЕВАЯ ГЕОМЕТРИЯ И РЕНДЕРИНГ

- `nodeHalfExtents(id)` — размеры через `canvas.measureText()` (НЕ DOM). Для групп: `n.width/2`, `n.height/2`. Для multi: `n.width/2`, `n.height/2`.
- `pillBorderPoint(cx,cy,hw,hh,dx,dy)` — аналитическое пересечение с pill-формой.
- `groupSnapPoint` — привязка линий к 4 кардинальным точкам групп.
- `render()` — двухпроходный: сначала группы → линии → узлы сверху.
- `bzAtDist(e, ti0, dist)` — точка на кривой на заданном расстоянии от границы.
- `lineExitFrom(e)` / `lineEntryTo(e)` — используют касательные к кривой.
- **Интра-групповые связи:** Линии между группой и узлом внутри неё. Отображаются без стрелки со стороны группы. Режим «прицепа» к рамке группы работает через `findNodeAt` и `getSnapPoint`.
- **Защита Root:** Корень (type='root') защищён от массового удаления. При удалении одиночного корня (через ПКМ) статус root автоматически переходит к orphans (потомкам без родителя).
- **НЕ использовать** `getBoundingClientRect()` — это архитектурное решение.

## 11. ВАЖНЫЕ АРХИТЕКТУРНЫЕ РЕШЕНИЯ (НЕ МЕНЯТЬ БЕЗ ОБСУЖДЕНИЯ)

1. **Измерение размеров узлов** через `canvas.measureText()` + CSS-константы, НЕ через `getBoundingClientRect()`.
2. **SVG (`#svgl`) позади узлов-div** (z-index). Кнопки на линиях видимы через JS-хендлеры.
3. **Корневой узел** — всегда `nodes[0]`, нельзя удалить через «удалить узел».
4. **Связи** (`isLink: true`) — декоративные, не влияют на иерархию (`gCh`, `gPar`, `isVisible`).
5. **Тач vs десктоп** определяется через `isMob() = window.innerWidth < 768 || 'ontouchstart' in window`.
6. **Файл всегда редактируется точечно**, не переписывается с нуля.
7. **Статус стартового узла (`type='root'`) неизменен:** `gPar()` в `utils.js` принудительно возвращает `null` для root-узла и игнорирует `link`-рёбра при вычислении иерархии. Ни одна линия не может сделать root-узел «дочерним».
8. **Fullscreen на Chrome/Wayland:** `requestFullscreen()` API технически срабатывает (события firing, элемент устанавливается), но Chrome на Wayland не разворачивает окно физически. F11 работает. Это системное ограничение Chrome/Linux, а не баг приложения. Решено оставить нативный вызов для совместимости с другими ОС/браузерами. Значок кнопки `⛶` намеренно не меняется.
9. **Рабочий URL:** `http://localhost:8080/`. `file:///` и другие порты — изолированное хранилище, не использовать для разработки.
10. **Динамический UI (Меню):** Плавающие меню используют `posMenu(el, x, y)` для центрирования и Clamp-логики. При открытии подменю (через `openSubmenu(id)`) вызывается принудительный ре-позиционинг родительского меню для адаптации к новой высоте блока, что гарантирует полную видимость всех настроек без скролла внутри скролла.

## 14. СТРАТЕГИЯ УПРАВЛЕНИЯ МОДЕЛЯМИ ИИ (РОУТИНГ)

Для эффективного и рационального использования выделенной квоты мощных ИИ-моделей (особенно на бесплатных тарифах) вводится обязательная стратегия коммуникации между агентом и пользователем. Система ИИ обязана управлять ожиданиями и заранее предупреждать о необходимой мощности перед тем, как приступить к написанию кода. Уровни сложности:

1. **Низкий (Low):** Верстка элементов, правка CSS-свойств, изменение цвета или текста, мелкие исправления багов UI без изменения логики (`Gemini 3 Flash` / `Gemini 3.1 Pro Low`).
2. **Средний (Medium):** Добавление стандартного UI-компонента (модалка, меню, выпадающий список), стандартный поверхностный рефакторинг JS-модулей, интеграция API без сложной обработки состояний (`Gemini 3 Flash` / `Gemini 3.1 Pro Low`).
3. **Высокий (High):** Капитальный рефакторинг архитектуры (как переход на File System Access API), сложная математика геометрии линий, работа с асинхронными состояниями и глубокими связями (graph theory) (`Gemini 3.1 Pro High` / `Claude Sonnet 4.6`).
4. **Экстремальный (Extreme):** Глубокая цикличная отладка, комплексное проектирование фундамента архитектуры, застой и неразрешимые баги (`Claude Opus 4.6` / `GPT-OSS 120B`).

**ОБЯЗАТЕЛЬНЫЕ ПРАВИЛА ДЛЯ АГЕНТА:**
- Агент должен в конце каждого сообщения при плановой работе указывать сложность *следующего* этапа и давать рекомендацию, какая модель ИИ потребуется (напр: "Нам предстоит работа с UI — можешь переключиться на Gemini 3 Flash").
- Если агент зашел в тупик (debugging loop) или предстоящая задача требует реструктуризации ядра, он **ДОЛЖЕН остановиться** и сообщить пользователю про необходимость переключения на модель `High` или `Extreme` перед следующим действием.
- Запрещено нерационально тратить "тяжелые" токены на попытки вслепую угадать проблему — лучше попросить переключить модель.
